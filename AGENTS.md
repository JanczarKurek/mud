# Repository Guidelines

## Project Structure & Module Organization
- `src/` holds the Bevy game code. Entry point: `src/main.rs`. 
- Files shouldn't be too long, systems should live in separate files (separate directories if they have to be splitted into several files)
- `target/` is generated by Cargo and should not be committed.
- There are no dedicated asset or test directories yet; add `assets/ if needed.
- File `ISSUES.md` contains list of features we would like to implement and problems we face. This file should be keep updated when something is fixed or when good idea occurs.

## Build, Test, and Development Commands
- `cargo run` — build and launch the game locally.
- `cargo build` — compile in debug mode without running.
- `cargo check` — fast type-checking without producing binaries. (Do this always after making any changes to the code before raporting success!)
- `cargo test` — run tests (none yet, but keep this green).
- `cargo fmt` — format code with rustfmt.
- `cargo clippy` — run lints; fix warnings before merging.

## Coding Style & Naming Conventions
- Use rustfmt defaults (4-space indentation, standard Rust formatting).
- Module files use `snake_case` names (e.g., `setup.rs`, `systems.rs`).
- Types in `PascalCase`, functions/vars in `snake_case`.
- Prefer small, focused Bevy systems; group related systems in `src/systems.rs`.

## Testing Guidelines
- No testing framework is set up yet.
- If you add tests, place unit tests in the module file with `#[cfg(test)]` and integration tests in `tests/`.
- Use `cargo test` for local verification and keep tests deterministic.

## Commit & Pull Request Guidelines
- Existing commits are short and informal; no strict convention is established.
- Use concise, imperative messages (e.g., “Add hex tile interaction”).
- PRs should include: summary, key files touched, and how to run/verify.
- Add screenshots or short clips for visual/interaction changes.

## Configuration Notes
- Rust toolchain is pinned via `rust-toolchain.toml`; match it when running CI or local builds.
- If a new crate is needed but the environment cannot download dependencies, update `Cargo.toml` first and ask the user to run a rebuild (`cargo build`/`cargo check`) so Cargo can fetch the crate. Continue implementation after the rebuild succeeds.
